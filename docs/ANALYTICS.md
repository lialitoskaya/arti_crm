# Аналитика

Раздел `Аналитика` показывает нагрузку по обращениям и скорость ответа менеджеров.

## Главное определение v87

**Обращение** — это первое входящее сообщение клиента в конкретном чате за локальный календарный день.

Пример:

```text
17.06 клиент написал 09:10, 09:12, 15:30 → считается 1 обращение за 17.06
18.06 тот же клиент написал 11:00, 11:05 → считается 1 обращение за 18.06
```

То есть повторные сообщения в том же чате в тот же день не увеличивают количество обращений. Но обращение того же клиента на следующий день учитывается снова.


## Среднее время ответа v87

Среднее время ответа больше не считается только по первому дневному обращению. Оно считается по каждому циклу ожидания ответа:

```text
10:58 клиент написал → 11:02 менеджер ответил = 4 минуты
11:09 клиент снова написал → 11:25 менеджер ответил = 16 минут
```

Если клиент написал несколько сообщений подряд до ответа менеджера, время считается от первого сообщения в этой цепочке.

```text
11:09 клиент написал
11:10 клиент дописал
11:25 менеджер ответил
= один диалоговый блок, 16 минут от 11:09
```

Если после сообщения клиента ответа менеджера нет, блок считается `без ответа` только если чат не находится в статусе `closed`. Закрытый чат считается завершённым и не требует ответа.

## Структура раздела

### 1. Верхние KPI

- **Всего обращений** — первые дневные обращения клиентов по чатам.
- **Клиентов / чатов** — уникальные клиенты и уникальные чаты за выбранный период.
- **Среднее время ответа** — по диалоговым блокам: первое входящее сообщение клиента после предыдущего ответа менеджера → ближайший следующий исходящий ответ менеджера. Если клиент написал несколько сообщений подряд до ответа, это один блок. Если чат уже в статусе `closed` и ответа нет, такой блок не считается ожидающим ответа.
- **До 10:00 / после 19:00** — первые дневные обращения, которые начались вне стандартного рабочего окна.
- **Закрытых обращений** — чаты со статусом `closed`, последняя активность которых попадает в выбранный период.
- **Пиковый час** — час, в который чаще всего случалось первое дневное обращение клиента.

### 2. Входящие сообщения

Сырые входящие сообщения тоже возвращаются API как `raw_inbound_messages` / `inbound_messages`, но они больше не являются главным счётчиком обращений.

Это нужно, чтобы видеть нагрузку по сообщениям отдельно от количества клиентских обращений.

### 3. Фильтры

- период дат;
- маркетплейс;
- диапазон часов.

Фильтр по часам применяется к часу первого дневного обращения клиента.

### 4. Пиковые часы

График по 24 часам показывает, в какое время чаще всего начинается дневное обращение клиента.

### 5. Обращения по дням

Таблица показывает по каждому дню:

- обращения — первые дневные обращения клиентов;
- сообщения — все входящие сообщения клиентов;
- уникальные чаты;
- закрытые чаты.

### 6. Разбивка по маркетплейсам

Показывает, где основная нагрузка: Ozon / WB / Яндекс.

## Timezone

Сообщения приходят из разных API в ISO/UTC-формате. Для группировки по локальным дням и часам используется:

```env
CRM_ANALYTICS_TZ_OFFSET_MINUTES=180
```

`180` = UTC+3.

## API

```text
GET /api/analytics/chats
```

Параметры:

```text
date_from=2026-06-01
date_to=2026-06-18
marketplace=ozon
hour_from=10
hour_to=18
tz_offset_minutes=180
```

Основные поля ответа:

```text
summary.requests                 # обращения по новой логике
summary.raw_inbound_messages     # все входящие сообщения
summary.avg_response_seconds     # среднее время ответа по диалоговым блокам
summary.response_blocks           # всего диалоговых блоков клиента
summary.answered_response_blocks  # блоки с ответом менеджера
summary.unanswered_response_blocks # блоки без ответа, кроме закрытых чатов
hourly[].requests                # пиковые часы по обращениям
daily[].requests                 # обращения по дням
daily[].inbound_messages         # входящие сообщения по дням
```

## Ограничения текущей версии

- Нет истории изменения статусов, поэтому `закрытые обращения` считаются по текущему статусу чата и последней активности.
- Нет SLA-таблицы, поэтому просрочки пока не выделяются отдельно.
- Среднее время ответа пока не исключает нерабочее время менеджеров.


## Служебные сообщения маркетплейсов

Начиная с v99 аналитика исключает сообщения, которые похожи на ботов, поддержку или системные события маркетплейсов. Такие сообщения остаются в истории диалога, но не попадают в KPI:
- обращения;
- входящие сообщения;
- пиковые часы;
- среднее время ответа.

Исключение выполняется по `author` и `raw_json`: `support`, `system`, `bot`, `notificationuser`, `systemuser`, `поддержка`, `система`, `бот` и похожие marketplace-маркеры.


## Drill-down по часам

Для проверки расчёта добавлен endpoint:

`GET /api/analytics/chats/drilldown`

Он возвращает:
- `items` — конкретные сообщения, которые входят в отчёт как первые обращения клиента за день;
- `hourly` — эти же строки, сгруппированные по часам;
- `excluded_service_items` — служебные сообщения, которые были исключены из KPI.
